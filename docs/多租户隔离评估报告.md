# GEOFlow 多租户隔离评估报告

> 评估对象：分支 `feature/multi-tenant-management`（提交 `6630b24 Add complete multi-tenant version`）
> 评估时间：2026-06-25
> 评估方式：源码静态审计（核心机制人工复核 + 5 路并行检索），未运行测试、未改动代码
> 关注重点：用户反馈"很多日志信息都没有隔离"，确认现状与"真正多租户"的差距

---

## 一、结论速览

**总体判断：当前实现是"单库 + 行级 `tenant_id` 隔离"的早期版本。同步 CRUD 主路径（23 个核心业务模型）隔离基本到位，但日志 / 分析 / 实时 / API 令牌 / 鉴权 / 文件存储 / 子表等"横切层"几乎完全没有隔离，且核心机制采用 fail-open（无租户上下文 = 看全部）设计，存在多处已确认的跨租户数据泄露与越权写入。距离生产可用的"真正多租户"还有明显差距。**

成熟度分层评估：

| 层面 | 隔离状态 | 说明 |
|------|---------|------|
| 同步业务 CRUD（文章/任务/知识库/作者/分类等 23 模型） | 🟢 基本到位 | 有 `tenant_id` 列 + `BelongsToTenant` trait，列表/读取/创建自动按租户过滤 |
| 后台异步任务（生成/趋势/主题克隆/URL导入） | 🟡 大部分到位 | 多数 Job 正确用 `TenantContext::run` 还原上下文，少数 Job 漏设上下文 |
| **日志 / 审计 / 分析（PV、任务运行、分发日志、活动日志）** | 🔴 **基本未隔离** | 相关表均无 `tenant_id`，大量用原生 `DB::table()` 读取，跨租户聚合 |
| **实时推送（Reverb / 任务进度广播）** | 🔴 **未隔离** | 单一全局频道，任意租户管理员可订阅他人任务总览 |
| **REST API / Sanctum 令牌** | 🔴 **弱** | 令牌不绑定租户、存在 IDOR、幂等键不分租户 |
| 鉴权 / 登录（用户名、锁定） | 🟡 全局命名空间 | 用户名全局唯一、锁定全局，跨租户可互相锁号 |
| 文件存储 / 缓存 | 🟡 部分 | 缓存键已按租户命名；文件路径未按租户隔离（靠随机文件名兜底） |
| 租户生命周期管理 | 🔴 缺失 | 无租户 CRUD 界面、无单租户多管理员、无租户停用拦截 |

**一句话差距：** 数据"写入归属"已建立，但"读取边界"在日志/分析/实时/API 层大面积失守，加上 fail-open + 超管 NULL 租户两处设计缺陷，使得"忘记设上下文"会静默泄露而非安全失败。

## 二、当前隔离机制（已有的基础）

1. **静态上下文** `App\Support\Tenancy\TenantContext`（`app/Support/Tenancy/TenantContext.php`）
   - 进程内静态变量保存 `tenantId` 与 `bypass`。
   - `shouldBypass()` 在 `bypass === true` **或 `tenantId === null`** 时返回 true —— 即**无租户即放行（fail-open）**。
   - `fromAdmin($admin)` 把 `bypass` 设为 `$admin->isSuperAdmin()` —— **超管完全绕过隔离**。
   - `run($tenantId, $cb)` 临时切换上下文并在 `finally` 还原，供后台任务使用。

2. **全局作用域** `App\Models\Scopes\TenantScope`（`TenantScope.php:14-18`）：非 bypass 时追加 `WHERE tenant_id = TenantContext::id()`。

3. **模型 trait** `App\Models\Concerns\BelongsToTenant`（`BelongsToTenant.php:10-21`）：boot 时挂全局作用域，并在 `creating` 时**仅当 `TenantContext::id()` 非空**才回填 `tenant_id`。

4. **三个上下文中间件**：
   - `InitializeTenantContext`（后台，从 `admin` guard 取）
   - `InitializeApiTenantContext`（API，从 `api_auth` 的 `auditAdminId` 反查 admin）
   - `InitializeSiteTenantContext`（前台，按请求域名匹配 `DistributionChannel.domain` 解析租户）

5. **数据库**：两个迁移给 25 张表加了 `tenant_id`（`2026_06_25_000000` 12 张 + `2026_06_25_010000` 13 张），并回填默认租户；超管的 `tenant_id` **故意留 NULL**。

> ⚠️ 一个模型只有在「有 `tenant_id` 列」**且**「使用 `BelongsToTenant` trait」时才真正隔离读取。只加列不加 trait = 看起来隔离、实际不隔离。

## 三、隔离现状矩阵

### 🟢 已完全隔离（有列 + trait，共 23 个模型）
`AiModel`、`Article`、`Author`、`Category`、`DistributionChannel`、`ImageLibrary`、`KeywordLibrary`、`KnowledgeBase`、`Prompt`、`Task`、`TitleLibrary`、`SiteSetting`、`KeywordTrendSource`、`KeywordTrendSourceSecret`、`KeywordTrendSnapshot`、`KeywordTrend`、`TopicPlan`、`TopicPlanItem`、`UrlImportJob`、`UrlImportJobLog`、`SiteThemeReplication`、`SiteThemeReplicationLog`、`SiteThemeReplicationVersion`。

> 复核结论：23 个 trait 模型对应的表全部有 `tenant_id`，**不存在"有 trait 无列"的破损情况**。

### 🟡 仅有列、无 trait（1 个，属有意为之）
- **`Admin`**（`app/Models/Admin.php`）：表有 `tenant_id`，但模型不挂 trait。这是租户锚点本身（登录、超管跨租户管理依赖它），有意不自动作用域。**残留风险**：任何直接 `Admin::query()` 都不会自动过滤，目前靠 `admin.super` 网关兜底。

### 🔴 完全未隔离（业务/敏感数据，无 `tenant_id`）

**(A) 隔离父模型的"子表"——可被直接查询绕过父级作用域：**

| 模型 | 表 | 父级 | 现状 |
|------|----|----|------|
| `ArticleDistribution` | article_distributions | Article/Channel | **确认泄露**（见 P1-④） |
| `DistributionLog` | distribution_logs | — | **确认泄露**（见 P1-④） |
| `TaskRun` | task_runs | Task | **确认泄露 + IDOR**（见 P1-②③） |
| `ArticleGeoAudit` | article_geo_audits | Article | **确认泄露**（见 P1-⑤） |
| `KnowledgeChunk` | knowledge_chunks | KnowledgeBase | 当前安全但脆弱（见 P2-①） |
| `ArticleReview` / `ArticleImage` / `TaskSchedule` / `DistributionChannelSecret` | — | Article/Task/Channel | 仅靠"先查到租户内父级再按父 id 过滤"间接安全，模型层无强制 |
| `Keyword` / `Title` / `Image` / `SensitiveWord` | keywords/titles/images/... | 各 Library | 多按 `library_id` 过滤间接安全；仪表盘全局计数除外 |

**(B) 日志 / 审计 / 分析 —— 用户重点关注，确认基本未隔离：**

| 数据 | 表 | 写入处 | 读取处（泄露点） |
|------|----|------|-----------------|
| 站点 PV / AI 爬虫访问 | view_logs（无 tenant_id） | `RecordSiteViewLog.php:33`（写入未带 tenant_id，即便此时已解析出租户） | `DashboardController.php:137/143`、`AnalyticsOverviewService.php:483/580`、`AnalyticsLogQueryService.php:64` 全用 `DB::table()` 跨租户聚合 |
| 任务运行记录 | task_runs | `JobQueueService.php:116`、`GeoFlowScheduleTasksCommand.php:54` | `AnalyticsOverviewService.php:330/496`、`DashboardController.php:192/201` 跨租户读取**失败任务的报错信息、任务名** |
| 分发日志 / 状态 | distribution_logs、article_distributions | `DistributionOrchestrator.php:236` | `DistributionController.php:58/54-55/315` 跨租户读全部发布历史、文章标题、渠道域名 |
| 管理员活动日志 | admin_activity_logs | `AdminActivityLogger.php:43`（无 tenant_id） | `AdminActivityLogController` 当前仅超管可读，暂未泄露，但**审计无租户归属** |
| API 幂等记录 | api_idempotency_keys | `IdempotencyService.php:88` | 唯一键仅 `(idempotency_key, route_key)`，**跨租户键冲突可重放他人响应**（见 P1-⑥） |

**(C) 平台/基础设施——无隔离属合理：**
`Tenant`（锚点）、`User`（Laravel 默认未用）、`SystemLog`（疑似死代码）、`SystemState`、`SystemUpdateRun/Backup`、`WorkerHeartbeat`、框架表（jobs/failed_jobs/cache/sessions/personal_access_tokens 等）。

## 四、确认的跨租户漏洞（按严重度，均带 file:line 证据）

> 前提澄清：本系统每个新管理员经 `TenantProvisioner` 自动分配独立租户，**普通管理员=某租户 owner（非超管）**。后台 `dashboard / analytics / distribution / geo-audit` 等路由只挂 `admin.auth + tenant.context`、**不挂 `admin.super`**，故以下"普通租户管理员可达"的泄露是真实可触发的。

### 🔴 P0 — Critical

**① 超管创建的记录 `tenant_id = NULL`，被所有 fail-open 读取路径全局可见**
- 链路：超管登录 → `fromAdmin` 设 `bypass=true` 且 `id()=NULL`（迁移故意不给超管回填租户，`create_tenants...:105-114`）→ 超管创建文章/任务时 `BelongsToTenant::creating` 因 `id()===null` **不回填 tenant_id**（`BelongsToTenant.php:17-19`）→ 记录落库 `tenant_id=NULL`。
- 后果：NULL 租户记录对任何 bypass/NULL 上下文读取者（超管、前台 fail-open、回退为超管的 API 令牌）全部可见，成为"租户孤儿数据"。
- 证据：`TenantContext.php:38`、`BelongsToTenant.php:17-19`、迁移 `:105-114`。

**② 前台公共站点在主域名/localhost/未匹配渠道域名时 fail-open，匿名用户可看到全部租户文章**
- `routes/web.php:45-53` 前台用 `site.tenant_context`；`InitializeSiteTenantContext.php:42-45` 对 `127.0.0.1/localhost`、以及任何未匹配 `DistributionChannel.domain` 的主机返回 null → 不设租户 → `Article/Category` 查询 fail-open。
- 后果：主域名/本地/未配置渠道的访问会把**所有租户的已发布文章（含 ①的 NULL 孤儿）混在一起渲染**给匿名用户。
- 证据：`InitializeSiteTenantContext.php:22-45`、`TenantScope.php:14`。

**③ 实时广播频道无租户隔离，任意租户管理员可订阅他人任务总览**
- `routes/channels.php:10-12`：`admin.tasks` 频道仅校验 `status==='active'`，**无 tenant_id 校验**，且是单一全局频道名。
- `TasksOverviewUpdated`（`app/Events/Admin/TasksOverviewUpdated.php:33-35`）向该共享频道广播完整任务总览；`recentRuns()`（`TaskMonitoringQueryService.php:429`）本身就跨租户取最近运行（含报错信息）。
- 后果：任一活跃管理员实时收到**其他租户的任务名、状态、报错**。
- 证据：`routes/channels.php:10`、`TasksOverviewUpdated.php:33-35`、`TaskMonitoringQueryService.php:429-447`。

### 🟠 P1 — High

**④ 分发后台：跨租户读 + 跨租户写 + 远程发布越权**
- 读：`DistributionController.php:58`（分发日志列表）、`:54-55`（待发/失败计数）、`:315`（任务列表）全部 `ArticleDistribution::query()/DistributionLog::query()` 无作用域（这两模型无 trait）。
- **写（更危险）**：`:491-493` `retry`、`:520-587` `editArticle/updateArticle/deleteArticle` 均以原生主键 `ArticleDistribution::query()->whereKey($id)` 取行，进而**编辑他人文章、向他人远程站点推送/删除内容、重新入队分发任务**。
- 证据：`DistributionController.php:54-587`、`DistributionOrchestrator.php:236`。

**⑤ GEO 审计跨租户读取**
- `GeoAuditController.php:28-70` 直接 `ArticleGeoAudit::query()`（无 trait）做 latest-ids/历史/详情查询，泄露所有租户文章的 GEO 评分、建议、风险备注。

**⑥ API 令牌不绑定租户 + 删除创建者后回退为超管（= 全租户）**
- 令牌只记 `tokenable=Admin`，无 tenant 字段；请求时有效租户 = 创建者 admin 当时的租户（隐式且可变）。
- `ApiTokenService.php:108-123` 的 `resolveAuditAdminId` 在创建者 id 为空/被删时**回退到最小 id 管理员（通常是种子超管）** → `bypass=true` → 该令牌可读写**全部租户**。
- 证据：`ApiTokenService.php:108-123`、`AuthenticateApiToken.php:40-44`、`InitializeApiTenantContext.php:19-26`。

**⑦ IDOR：`GET /api/v1/jobs/{job}` 可越租户读任意任务运行记录**
- `routes/api.php:58-60`（scope `jobs:read`）→ `JobController::show` → `TaskLifecycleService::getJob()`（`:395` `TaskRun::query()->find($jobId)`）**无归属校验**，`TaskRun` 又无租户作用域。
- 注：`/tasks/{task}/jobs` 因先 `ensureTaskExists`（已作用域）而安全。

**⑧ 幂等键跨租户冲突 → 重放他人响应 / 409 DoS**
- `IdempotencyService.php:54-110` 仅按 `(idempotency_key, route_key)` 命中；两租户用相同 `X-Idempotency-Key`（如 `create-1`）+ 相同 body → B 拿到 **A 的缓存响应**；不同 body → 409 阻断 B 的正常写入。

**⑨ 用户名全局唯一 + 登录锁定全局 → 跨租户锁号 DoS**
- `admins.username` 全局唯一（建表迁移 `:18`）；登录仅按用户名查（`AdminAuthController.php:48`、`ApiAdminAuthService.php:50`）；锁定按用户名写共享缓存并置 `status=locked`（`AdminLoginLockService.php:51/73-76`）。知道任意管理员用户名即可跨租户锁其账号。

**⑩ 仪表盘 / 分析跨租户聚合（PV、任务、知识库/词库计数）**
- 见三-(B)。`view_logs`、`task_runs` 经原生 `DB::table()` 读取，普通租户管理员的仪表盘/分析页展示的是**全平台**流量与任务指标，并能看到他人失败任务报错。

### 🟡 P2 — Medium

**① `KnowledgeChunk`（RAG 向量）当前安全但脆弱**：无 tenant_id/trait，但召回查询（`KnowledgeRetrievalService.php:704-715`、`WorkerExecutionService.php:1168-1179`）与 20+ 处直查均按 `knowledge_base_id`（源于已作用域的 KB）过滤 → **未发现内容泄露**；仅后台聚合计数为全局。一旦有人写 `KnowledgeChunk::find()` 或去掉父级判空即泄露。

**② `ProcessArticleDistributionJob` 未设租户上下文**：`ProcessArticleDistributionJob.php:24` 直接按主键加载 `ArticleDistribution` 并走 orchestrator，期间 `Article/DistributionChannel`（租户模型）在 NULL 上下文下 fail-open 查询。当前靠主键驱动可用，但破坏隔离不变式，应仿照其他 Job 包 `TenantContext::run`。

**③ `OptimizeArticleGeoJob` 未设租户上下文**：`OptimizeArticleGeoJob.php:40-43` 以 id `find` 文章，fail-open。

**④ 已发布主题目录可能跨租户覆盖**：`ThemeReplicationPublishService.php:41` 写共享 `public_path("themes/{theme_id}")`；有 `targetExistsForAnotherTheme` 守卫，但 `theme_id` 非按租户全局唯一时仍有碰撞/覆盖风险。

**⑤ 管理员活动日志无租户归属**：当前仅超管可读未泄露，但破坏"按租户审计"能力，且一旦放开读取入口即泄露。

### 🟢 P3 — Low / 设计可接受

- 文件存储未按租户分目录（图片库、KB 上传、编辑器资源走共享 `uploads/...` + 随机文件名兜底）：`config/geoflow.php:117-119`、`ImageLibraryController.php:450-458` 等。实际泄露概率低，但无文件系统级隔离。
- `geoflow:fetch-keyword-trends --sync` 分支未设上下文（仅运维手动触发；定时异步路径安全）。
- `system-updates` 路由未挂 `admin.super`（更新针对单一共享部署，跨租户可见尚属合理，但建议收敛到超管）。
- 缓存键已正确按租户命名（`SiteSettingsBag.php:59-62`，`run` 内验证），**未发现缓存投毒**。
- 登录锁定虽全局，但因用户名全局唯一，技术上仍是按账号（⑨从 DoS 角度仍列 High）。

## 五、根因分析（设计层面，比逐点修复更重要）

1. **Fail-open 默认放行**：`shouldBypass()` 把 `tenantId === null` 等价于"看全部"。任何"忘记设上下文"的路径都会**静默泄露**而非报错/查空。真正多租户应 **fail-closed**（无上下文默认查不到数据，仅显式 bypass 才放行）。

2. **超管 `tenant_id = NULL` + 创建时不回填**：超管写入产生 NULL 孤儿数据，与 fail-open 叠加放大泄露面。超管应有"代某租户操作"的显式上下文，或创建时强制指定归属租户。

3. **静态进程级上下文**：依赖每条执行路径手动 set/clear，遗漏即出错；队列、命令、广播回调等非请求路径尤其脆弱。

4. **大量原生 `DB::table()` 绕过全局作用域**：分析/仪表盘/监控层即便模型挂了 trait 也会被绕过（`AnalyticsLogQueryService`、`AnalyticsOverviewService`、`TaskMonitoringQueryService`、`DashboardController`）。需在原生查询里显式补 `tenant_id` 条件。

5. **日志/事件/审计/分析表整体漏加 `tenant_id`**：两个迁移只覆盖业务主表，把所有 `*_logs`/PV/运行记录/幂等键排除在外 —— 这正是用户反馈的核心。

6. **子表未随父表隔离**：`ArticleDistribution/DistributionLog/TaskRun/ArticleGeoAudit/ArticleReview/...` 可被直接查询绕过父级作用域。

7. **鉴权与基础设施全局命名空间**：用户名全局唯一、登录锁定全局、API 令牌不绑租户、幂等键不分租户。

8. **租户生命周期能力缺失**：无租户管理 UI、单租户仅单管理员（`AdminUserController::store` 每个新管理员都新建租户，无法把人加入既有租户）、无 `Tenant.status` 停用拦截。

## 六、与"真正多租户"的差距清单（分领域 + 建议）

**数据层**
- [ ] 为 `ArticleDistribution`、`DistributionLog`、`TaskRun`、`ArticleGeoAudit`、`ArticleReview`、`TaskSchedule`、`KnowledgeChunk` 增加 `tenant_id` + `BelongsToTenant`（或全部改为经已作用域父级查询，禁用按裸 id 查子表）。
- [ ] 复核所有 `withoutGlobalScopes()` 使用点，确认仅限后台自解析租户。

**日志 / 审计 / 分析（用户重点）**
- [ ] `view_logs`、`task_runs`、`distribution_logs`、`admin_activity_logs`、`api_idempotency_keys` 增加 `tenant_id`，写入处补齐归属（`RecordSiteViewLog`、`AdminActivityLogger`、`DistributionOrchestrator`、`JobQueueService`）。
- [ ] 重写 `AnalyticsOverviewService` / `AnalyticsLogQueryService` / `TaskMonitoringQueryService` / `DashboardController` 中所有原生 `DB::table()`，显式补租户条件。
- [ ] 幂等键唯一约束改为 `(tenant_id, idempotency_key, route_key)`。

**异步 / 后台**
- [ ] `ProcessArticleDistributionJob`、`OptimizeArticleGeoJob` 包 `TenantContext::run`（参照 `ProcessGeoFlowTaskJob` 的标量 `tenant_id` 还原模式）。
- [ ] `geoflow:fetch-keyword-trends --sync` 分支补上下文。

**实时**
- [ ] 频道改为按租户命名（如 `admin.tasks.tenant.{id}`），授权回调校验 `admin.tenant_id === id`（或超管）；广播定向到发起租户频道；`recentRuns()` 按租户过滤。

**API / 鉴权**
